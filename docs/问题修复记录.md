## 批次索引映射Bug修复

### 问题概述

**发现时间**: 2026-02-21  
**严重程度**: 🔴 严重 (导致88%文章未被AI处理)  
**影响范围**: AI筛选模块,所有批次(除第1批)

### 问题描述

从第2批开始,大量文章被标记为"未解析"并跳过AI筛选,导致只有第1批的文章能正常处理。

**症状**:
```
批次1 (文章1-15):  ✅ 正常处理
批次2 (文章16-30): ❌ 全部"未解析"
批次3 (文章31-45): ❌ 全部"未解析"
...
```

**数据**:
- 总文章数: 124篇
- 应处理: 124篇
- 实际处理: 15篇 (12%)
- 未解析: 109篇 (88%)
- 识别率: 仅6.5%

### 根本原因分析

#### 初始错误诊断 ❌

最初误认为问题是:
- QPS限流导致后续批次失败
- AI上下文不足处理15篇文章
- 异步请求之间的竞争条件

**采取的错误措施**:
1. 将BATCH_SIZE从15减到10
2. 将"未解析"默认标记为"不相关"
3. 过度强化prompt

**用户纠正**:
> "不对.我使用的LLM API 上下文足以处理批处理15篇内容的.之前大量的'未解析'是不是因为批处理之间采用了异步方式请求?我的LLM API 上下文足够长,但可能存在QPS限制,因此需要帮我调整批间处理逻辑,确保不同批都被LLM API 处理到.而不是简单的将'未解析'改成'不相关',这样会造成误导"

#### 真正的根本原因 ✅

**批次索引映射错误**

**问题代码** (`llm_filter.py:244-252`):

```python
# 错误的逻辑
ai_judgments = {}
for r in results:
    idx = r.get('index', 0) - 1  # AI返回16,减1得15
    if 0 <= idx < len(articles):  # len(batch)=15,条件失败! (15不在0-14范围)
        ai_judgments[idx] = r  # 永远不会执行
```

**问题分析**:

| 批次 | AI返回索引 | 减1后索引 | 批次长度 | 范围检查 | 结果 |
|------|-----------|----------|---------|---------|------|
| 批次1 | 1-15 | 0-14 | 15 | 0-14在0-14内 | ✅ 通过(碰巧) |
| 批次2 | 16-30 | 15-29 | 15 | 15-29不在0-14内 | ❌ 失败 |
| 批次3 | 31-45 | 30-44 | 15 | 30-44不在0-14内 | ❌ 失败 |

**核心问题**: AI返回的是全局索引(16-30),但代码检查的是批次内索引(0-14)范围,导致索引映射失败。

### 解决方案

#### 修复代码

**文件**: `/root/project-wb/n8n/src/llm_filter.py`

**修改1 - 函数签名** (Line 142):

```python
# 修改前
def parse_batch_result(result: str, articles: List[Dict]) -> Tuple[List[Dict], List[Dict]]:

# 修改后
def parse_batch_result(result: str, articles: List[Dict], start_index: int = 1) -> Tuple[List[Dict], List[Dict]]:
```

**修改2 - 索引映射逻辑** (Lines 244-252):

```python
# 修改前 (错误)
ai_judgments = {}
for r in results:
    idx = r.get('index', 0) - 1  # AI返回16,减1得15
    if 0 <= idx < len(articles):  # 15不在0-14范围内!
        ai_judgments[idx] = r

# 修改后 (正确)
ai_judgments = {}
for r in results:
    global_idx = r.get('index', 0)  # AI返回的全局索引(如16)
    relative_idx = global_idx - start_index  # 转换为批次内索引: 16-16=0 ✅
    if 0 <= relative_idx < len(articles):
        ai_judgments[relative_idx] = r
    else:
        print(f"  ⚠ 索引超出范围: global_idx={global_idx}, relative_idx={relative_idx}")
```

**修改3 - 调用点更新** (Line 333):

```python
# 修改前
relevant, failed = parse_batch_result(result, batch)

# 修改后
relevant, failed = parse_batch_result(result, batch, i + 1)  # 传入起始索引
```

**修改4 - 回滚错误修改**:

```python
# 回滚1: 恢复BATCH_SIZE

# 回滚2: 移除"默认不相关"的错误逻辑

# 回滚3: 简化prompt
# 移除了过度复杂的prompt强化
```

## 去重机制验证

### 验证概述

**验证时间**: 2026-02-21  
**验证目的**: 确认已推送文章不会重复推送  
**验证方法**: 代码审查 + 功能测试 + 数据完整性检查

### 去重机制架构

**核心流程**:

```
RSS文章获取 (195篇)
    ↓
生成文章哈希 (MD5: link优先,降级为title+source)
    ↓
查询数据库 (is_article_pushed)
    ↓
过滤已推送 (filter_new_articles) - 过滤掉53篇
    ↓
新文章 (142篇) → AI筛选
    ↓
推送到企业微信 (80篇相关)
    ↓
标记为已推送 (mark_articles_as_pushed)
    ↓
写入SQLite数据库
```

### 哈希生成策略

**优先级**:
1. 优先使用链接 (最可靠)
2. 降级使用标题+来源 (无链接时)

**代码**: `/root/project-wb/n8n/src/auto_push.py`

```python
def get_article_hash(article):
    """生成文章唯一哈希（基于链接或标题）"""
    unique_str = article.get('link', '') or f"{article.get('title', '')}_{article.get('source_name', '')}"
    return hashlib.md5(unique_str.encode('utf-8')).hexdigest()
```

### 数据库设计

**表结构**:

```sql
CREATE TABLE IF NOT EXISTS push_history (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    article_hash TEXT UNIQUE NOT NULL,  -- 唯一约束
    title TEXT,
    link TEXT,
    source_name TEXT,
    push_time TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

CREATE INDEX idx_article_hash ON push_history(article_hash);
CREATE INDEX idx_push_time ON push_history(push_time);
```

**关键特性**:
- ✅ `UNIQUE`约束防止重复插入
- ✅ 索引优化查询性能(<1ms/篇)
