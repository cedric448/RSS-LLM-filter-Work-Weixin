## 2. RSS重试机制详解

### 2.1 配置参数

```python
RSS_FETCH_TIMEOUT = 15      # 单次请求超时(秒)
RSS_MAX_RETRIES = 2          # 最大重试次数
RSS_RETRY_DELAY = 3          # 基础重试间隔(秒)
```

### 2.2 重试策略

**指数退避算法**:
```
第1次重试延迟: 3秒  (3 × 2^0)
第2次重试延迟: 6秒  (3 × 2^1)
```

**触发条件**:
- ✅ `requests.exceptions.Timeout` - 请求超时
- ✅ `requests.exceptions.ConnectionError` - 连接失败
- ❌ 其他异常 - 不重试(如XML解析错误)

### 2.3 工作流程

```
第1次尝试
  ↓ 失败(超时/连接)
⏳ 等待3秒
  ↓
第2次尝试(重试1/2)
  ↓ 失败
⏳ 等待6秒
  ↓
第3次尝试(重试2/2)
  ↓ 失败
❌ 最终失败
```

### 2.4 输出示例

```bash
# 首次失败
  ⏳ 卡尔的AI沃茨: 连接失败, 3秒后重试 (1/2)

# 重试成功
  ✓ 卡尔的AI沃茨: 5 篇文章 (重试1次后成功)

# 重试全部失败
  ✗ 某源名称: 重试2次后仍失败
```

### 2.5 性能影响

**时间成本**:
- 单源最坏情况: 15秒 + 3秒 + 15秒 + 6秒 + 15秒 = **54秒**
- 10并发情况: 由于并发执行,总体影响较小
- 实测: 40源从3分钟增加到3.2分钟(增加约6%)

**成功率提升**:
- 优化前: 45% (18/40)
- 优化后: **100%** (40/40) ✅

## 3. 分批推送详解

### 3.1 配置参数

```python
MAX_ARTICLES_PER_BATCH = 20  # 单批最大文章数
BATCH_INTERVAL = 2           # 批次间延迟(秒)
MAX_MESSAGE_BYTES = 4000     # 单条消息最大字节数
```

### 3.2 分批策略

**双重检查机制**:

1. **数量检查** (优先级高):
   - 累积文章数 ≥ 20篇 → 立即分批

2. **字节检查** (精确控制):
   - 构建测试消息
   - 字节数 > 4000 → 移除最后一篇,分批

**智能处理**:
- 单篇过长 → 简化格式(移除摘要)
- 批次间等待 → 避免频繁请求

### 3.3 分批示例

#### 示例1: 25篇文章

```bash

  → 将分 2 批推送 (每批最多 20 篇)

  消息长度: 1646 字符 / 2503 字节
✓ 消息推送成功
  ⏳ 等待 2 秒后发送下一批...

📦 正在推送第 2/2 批(5 篇文章)
  消息长度: 475 字符 / 716 字节
✓ 消息推送成功
```

#### 示例2: 文章总数显示

推送消息中会显示:

### 3.4 消息格式

**完整格式**:

## 6. 错误处理

### 6.1 RSS获取错误

**可重试错误**:
```python
✅ requests.exceptions.Timeout      # 请求超时
✅ requests.exceptions.ConnectionError  # 连接失败
```

**不可重试错误**:
```python
❌ XML解析错误                      # 数据格式问题
❌ HTTP 404/500等状态码错误         # 服务端问题
❌ 其他未知异常                     # 避免无限重试
```

### 6.2 推送错误

**自动处理**:
- 单批失败不影响其他批次
- 消息过长自动截断
- UTF-8编码安全处理

**失败处理**:
```python
if not success:
    print(f"✗ 第 {batch_num} 批推送失败")
    all_success = False
    # 继续推送后续批次
```

## 10. 故障排查

### 10.1 重试全部失败

**现象**:
```
✗ 某源: 重试2次后仍失败
```

**可能原因**:
1. RSS源长期不可用
2. 网络连接问题
3. RSS服务端限流

**解决方案**:
1. 检查RSS源URL是否有效
2. 增加重试次数: `RSS_MAX_RETRIES = 3`
3. 延长超时时间: `RSS_FETCH_TIMEOUT = 20`
4. 临时禁用该源

### 10.2 分批推送失败

**现象**:
```
✗ 第 1 批推送失败
```

**可能原因**:
1. 企业微信Webhook失效
2. 消息格式错误
3. 网络问题

**解决方案**:
1. 测试Webhook: `python3 src/wechat_pusher.py`
2. 检查消息字节数是否超限
3. 检查网络连接

### 10.3 消息截断

**现象**:
```
⚠️ 消息超长(按字节),强制截断
```

**原因**: 单批消息超过4096字节

**解决方案**:
1. 降低每批文章数: `MAX_ARTICLES_PER_BATCH = 15`
2. 简化消息格式(已自动处理)
3. 缩短文章标题长度
