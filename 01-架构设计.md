# 系统架构设计文档

## 1. 系统概述

RSS 智能筛选推送系统是一个基于 Python + Cron + AI 大模型的自动化内容分发系统,通过今天看啥 API4 接口获取公众号文章,经大模型进行 AI 主题筛选,并推送到企业微信。

### 1.1 技术栈

| 组件 | 技术选型 | 版本 | 用途 |
|------|---------|------|------|
| 数据库 | SQLite | - | 去重存储 |
| AI 模型 | kimi-k2.5-ioa | - | 内容筛选 |
| 语言 | Python | 3.11+ | 核心业务逻辑 |
| 定时任务 | Crontab | - | 定时触发 |

### 1.2 核心特性

- ✅ **自动化推送**: 定时 (9:00/13:00/20:00) 自动执行
- ✅ **智能去重**: 基于 SQLite 的文章哈希去重
- ✅ **AI 筛选**: 语义理解,非关键词匹配
- ✅ **多源聚合**: 支持多个公众号 RSS 源
- ✅ **容错机制**: 超时重试、限流避让、错误处理
- ✅ **统计监控**: 推送历史、成功率统计

## 3. 核心模块设计

### 3.1 模块划分

```
src/
├── auto_push.py       # 主入口 - 编排整体流程
├── llm_filter.py      # AI筛选模块 - 调用大模型API
└── wechat_pusher.py   # 推送模块 - 企业微信消息发送
```

### 3.2 模块职责

#### 3.2.1 auto_push.py (主控模块)

**职责**:
- 定时任务入口
- API4 文章获取
- 去重数据库管理
- 流程编排

**核心函数**:
```python
- init_database()              # 初始化SQLite去重库
- get_rss_sources()            # 读取RSS源配置
- fetch_articles_via_api4()    # API4方式获取文章
- is_article_pushed()          # 检查是否已推送
- mark_articles_as_pushed()    # 标记为已推送
- clean_old_history()          # 清理历史记录
```

**容错设计**:
- 超时重试: 3次指数退避 (3s/6s/12s)
- 限流检测: 识别404/503限流响应
- 并发控制: 最大3并发,源多时自动降级顺序获取
- 请求间隔: 1秒间隔避免限流

#### 3.2.2 llm_filter.py (AI筛选模块)

**职责**:
- 构建AI筛选提示词
- 调用大模型API
- 解析AI返回结果
- 置信度过滤

**核心函数**:
```python
- filter_articles_batch()      # 批量筛选文章
- build_prompt()               # 构建提示词
- parse_ai_response()          # 解析AI响应
```

**筛选逻辑**:
1. 读取 `config/keywords.json` 配置
2. 构建语义筛选提示词
3. 批量调用大模型API (每批15篇)
4. 解析返回的JSON结果
5. 过滤置信度 > 0.7 的文章

#### 3.2.3 wechat_pusher.py (推送模块)

**职责**:
- 格式化Markdown消息
- 调用企业微信Webhook
- 推送结果处理

**核心函数**:
```python
- push_articles()              # 推送文章列表
- format_message()             # 格式化消息
- send_to_wechat()             # 发送请求
```

**消息格式**:
```markdown
📰 **{来源} | {分类}**
[{标题}]({链接})

💡 推荐理由: {AI生成的理由}
```

### 3.3 数据库设计

#### 3.3.1 去重数据库 (SQLite)

**表结构**:
```sql
CREATE TABLE push_history (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    article_hash TEXT UNIQUE NOT NULL,      -- MD5(link or title+source)
    title TEXT,
    link TEXT,
    source_name TEXT,
    push_time TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

CREATE INDEX idx_article_hash ON push_history(article_hash);
CREATE INDEX idx_push_time ON push_history(push_time);
```

**去重策略**:
- 优先使用文章链接生成哈希
- 链接为空时使用 `标题+来源` 生成哈希
- 哈希唯一约束避免重复插入

**清理策略**:
- 每次运行时自动清理 30 天前的历史记录
- 防止数据库无限增长

## 6. 性能指标

### 6.1 关键性能参数

| 指标 | 数值 | 说明 |
|------|------|------|
| RSS源数量 | <200 | 公众号订阅数 |
| 每源文章数 | 20 | 每个源最多获取20篇 |
| 并发获取数 | 3 | 最大并发RSS请求数 |
| 请求间隔 | 1秒 | 避免限流 |
| 超时时间 | 15秒 | API4获取超时 |
| 重试次数 | 3次 | 失败重试次数 |
| 去重保留期 | 30天 | 历史记录保留 |
| AI批处理 | 15篇 | 每批次文章数 |
| 置信度阈值 | 0.7 | AI筛选阈值 |

### 6.2 预期性能

- **全流程耗时**: 约 2-5 分钟 (取决于源数量和文章数)
- **文章获取**: 顺序获取 ~1秒/源, 并发获取 ~5秒/10源
- **AI筛选**: ~5秒/批 (15篇)
- **推送**: <1秒

## 8. 技术亮点

### 8.1 容错机制

1. **超时重试**: 指数退避策略 (3s/6s/12s)
2. **限流检测**: 自动识别并重试限流响应
3. **并发降级**: 源多时自动切换顺序获取
4. **错误隔离**: 单个源失败不影响其他源

### 8.2 性能优化

1. **并发获取**: ThreadPoolExecutor 并发抓取
2. **批量处理**: AI筛选批量调用,减少API开销
3. **智能去重**: 哈希索引快速查询
4. **定期清理**: 自动清理历史数据

### 8.3 可扩展性

1. **配置驱动**: RSS源和筛选规则配置化
2. **模块解耦**: 获取/筛选/推送独立模块
3. **API4 接口**: 今天看啥 API4 JSON 接口获取文章
